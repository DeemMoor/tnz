# Roadmap: сайт турниров по настольному теннису (nz)

> Стек: PHP 8.5, **Symfony 7.4 LTS** (backend, JSON API), **React + Vite** (публичный фронт), **EasyAdmin** (админка, серверная, без React), MySQL 8.
> Хостинг: **BeGet shared** (как jegebce.online), деплой по SSH (`git pull` + `scripts/deploy-remote.sh`).
> Формат работы: задачи мелкие, ~на один запуск. Каждая задача = отдельный коммит + запись в `docs/devlog/` (см. CLAUDE.md).

## Согласованные решения

- **Auth:** регистрация и вход по **телефону + пароль**. SMS-подтверждения нет (слой можно добавить позже).
- **Email:** в личном кабинете можно добавить email и подтвердить его — **письмом по SMTP** (Symfony Mailer).
- **Walk-in:** админ может **завести игрока «с колёс»** прямо на чекине (телефон + имя) и сразу его чекинить.
- **Жеребьёвка:** одна кнопка у админа — рандом. Система делит чекнувшихся на 2 стола по 16 и внутри стола случайно составляет пары 1-го тура. При нехватке до 16 — авто-проход (bye).
- **Сетка:** single-elimination на каждом столе: 1/16 → 1/4 → 1/2 → финал. Победитель стола → `isChampion`, в обычных турнирах больше не участвует.
- **Кто отмечает результат:** победителя матча может нажать **админ ИЛИ любой из двух игроков этого матча** — тапнул победившего, результат отобразился в сетке.
- **Mobile-first:** основная аудитория с телефонов (регистрация, чекин, отметка результата, просмотр прогресса). Весь фронт верстаем сначала под мобилу.
- **RTTF-рейтинг:** в профиле игрок сам указывает рейтинг с rttf.ru (или «нет рейтинга»). Доверяем без проверки. **Гейт:** рейтинг строго выше 250 → регистрироваться на турнир нельзя (турнир для любителей ≤250 и без рейтинга).
- **Внутренний рейтинг / статистика:** очки = +1 за каждую победу в матче. Рейтинг игрока = суммарное число побед за всё время. Страница статистики (сыграно/побед/проигрышей/очки) — **публичная**, данные считаются из матчей.
- **Нумерация турниров:** первый турнир (#1) — **15 марта 2026** (воскресенье), далее еженедельно. Номер = недель от якоря + 1 (`TournamentSchedule::number`, константа `FIRST_TOURNAMENT_DATE`). Отображение: «Турнир #19 · 19 июля 2026». Пример: 19.07.2026 = **#19**.

### Жизненный цикл регистрации на турнир (важно)

- **Турниры еженедельные**, все сохраняются (история). Админ создаёт турнир, задавая **дату (воскресенье)**. Времена **зашиты в код** и считаются от этой даты.
- **Расписание (хардкод):**
  - Регистрация открывается **в четверг перед турниром в 16:00**.
  - Окно чекина — **в воскресенье 14:00–14:15**.
- **Вместимость — 32** (2 стола × 16). Первые 32 зарегистрировавшихся → участники (`registered`), остальные → **очередь ожидания** (`waitlisted`, порядок по времени регистрации).
- **Снять с регистрации** (кнопка в кабинете, до турнира): место освобождается, **первый из очереди автоматически поднимается** в участники.
- **Чекин** вс 14:00–14:15: участник отмечает присутствие (или админ). 
- **В 14:15 админ жмёт «Закрыть чекин»:** все `registered`, кто **не** зачекинился, → `dropped`; освободившиеся места **добираются из очереди по порядку — первому в очереди, независимо от того, зачекинился ли он** (так решено). Итоговый ростер (`registered` после обработки) идёт в жеребьёвку.
- **Автоматизация — гибрид:** открытие регистрации проверяется **сравнением с текущим временем** (без cron); сброс но-шоу в 14:15 — **кнопкой админа** (он и так на месте). Cron не нужен.

## Модель данных

| Сущность | Поля (черновик) |
|---|---|
| `User` | id, phone (unique), password (хеш), name, email (nullable), emailVerified, emailVerificationToken (nullable), rttfRating (nullable int, null = нет рейтинга), roles (json), isChampion, createdAt |
| `Tournament` | id, name, **date** (дата воскресенья), status (enum: draft / registration / checkin / drawn / in_progress / finished), createdAt. Окна регистрации/чекина не хранятся — считаются от `date` (см. `TournamentSchedule`) |
| `TournamentEntry` | id, tournament, user, **status** (registered / waitlisted / cancelled / dropped), **registeredAt** (для порядка в основе и очереди), checkedIn (bool), checkedInAt (nullable), tableNumber (1/2, nullable до жеребьёвки), eliminated. Unique (tournament, user) |
| `Match` | id, tournament, tableNumber, round (1=1/16 … 4=финал), slot (позиция в туре), player1 (nullable), player2 (nullable), winner (nullable), status (pending/done), playedAt (nullable) |

- Статусы турнира: `draft → registration → checkin → drawn → in_progress → finished`.
- Вместимость `CAPACITY = 32` — константа. Порядок в основе и в очереди — по `registeredAt` asc.

## Хостинг: BeGet shared

- **Один docroot.** Код Symfony живёт выше веб-корня, его `public/` назначается каталогом сайта (`public_html`).
- **SPA + API на одном домене.** React собирается в статику (`npm run build`) в `public/` Symfony; fallback-роут отдаёт `index.html` на все не-`/api` пути. Один домен → нет CORS, сессионная авторизация работает искаропки. EasyAdmin на `/admin`.
- **Нет долгоживущих процессов.** API — обычный запрос-ответ через PHP-FPM. Vite dev-сервер только локально.
- **PHP-настройки** через `.htaccess` (`php_value ...`) — `memory_limit`, `upload_max_filesize`.
- **SMTP** для верификации email — почта BeGet (креды в `.env.local`).

### T3 — Турниры и регистрация
- [x] T3.1 Сущности `Tournament` (date=вс, status), `TournamentEntry` (status, registeredAt, checkedIn…), миграция.
- [x] T3.2 Сервис `TournamentSchedule` — из `date` считает окна: регистрация с чт 16:00, чекин вс 14:00–14:15 (времена хардкод-константы). Хелперы `isRegistrationOpen(now)`, `isCheckinOpen(now)`.
- [x] T3.3 EasyAdmin CRUD турниров (создание с датой, смена статуса, просмотр записей).
- [x] T3.4 `POST /api/tournaments/{id}/register` — регистрация. Гейты: окно регистрации открыто; не чемпион; не дубль; **RTTF > 250 нельзя** (403). Первые `CAPACITY=32` → `registered`, дальше → `waitlisted` (по `registeredAt`).
- [x] T3.5 `DELETE /api/tournaments/{id}/registration` — «Снять с регистрации»: `cancelled`; если освободилось место из основы — **автопромоут первого из очереди** в `registered`.
- [x] T3.6 Frontend: список турниров + ближайший; кнопка «Участвовать»/«Снять с регистрации»; показ статуса (участник / очередь #N); подсказка когда откроется регистрация.

### T4 — Чекин и обработка очереди
- [x] T4.1 `POST /api/tournaments/{id}/checkin` — игрок сам отмечается (в окне вс 14:00–14:15; только `registered`). Плюс админский вариант `.../checkin/{userId}`.
- [x] T4.2 Walk-in: `POST /api/admin/tournaments/{id}/walk-in` — завести игрока (телефон+имя, авто-пароль) и сразу чекинить (если есть место / из очереди).
- [x] T4.3 `POST /api/admin/tournaments/{id}/close-checkin` (кнопка 14:15): `registered && !checkedIn` → `dropped`; добор мест из очереди по порядку (первому в очереди, независимо от чекина); статус турнира → `checkin`/готов к жеребьёвке.
- [x] T4.4 Экран чекина у админа: списки основы/зачекиненных/очереди, счётчик 0/32, кнопка «Закрыть чекин».

### T5 — Жеребьёвка

Примеры: 32 → 16/16; 20 → 16/4; 12 → 12/0. Внутри стола генерит матчи 1/16 (+ bye при нехватке до степени двойки).
