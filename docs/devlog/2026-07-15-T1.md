# T1 — Аутентификация (2026-07-15)

## Что сделал

**Backend:**
- Сущность `User` (`src/Entity/User.php`): телефон (unique), хеш пароля, имя, email (nullable) + `emailVerified`/`emailVerificationToken` (задел под T2), roles, `isChampion`, `createdAt`. Реализует `UserInterface`/`PasswordAuthenticatedUserInterface`, идентификатор для Security — телефон.
- `UserRepository` с `findOneByPhone()` и `upgradePassword()` (прозрачное обновление хеша при входе).
- Миграция `Version20260715073511` — таблица `user` с unique-индексом на `phone`.
- `PhoneNormalizer` — приводит телефон к виду `7XXXXXXXXXX` (понимает `+7`, `8`, `10 цифр`, скобки/пробелы). Так уникальность работает независимо от формата ввода.
- `POST /api/register` — регистрация: валидация (телефон/пароль ≥6/имя), нормализация, проверка дубля (409), хеш пароля (argon2id), 201 с базовыми данными.
- Security (`config/packages/security.yaml`): провайдер из БД по телефону, firewall с `json_login` (сессия, cookie PHPSESSID), `logout`. `access_control`: публичные `login/register/ping/verify-email`, `/admin` → `ROLE_ADMIN`, `/api/me` → `ROLE_USER`.
- `SecurityController`: `POST /api/login`, `POST /api/logout`, `GET /api/me`.
- Тесты `tests/Controller/AuthTest.php` — 6 функциональных тестов (регистрация/дубль/валидация/me-401/login+me/неверный пароль). **Все зелёные.**

**Frontend:**
- `AuthContext` (контекст + хук `useAuth`) — «раздаёт» текущего пользователя во всё приложение, методы `login/logout/refresh`. При старте один раз проверяет сессию через `/api/me`.
- `ProtectedRoute` — guard: пускает на приватную страницу только залогиненного, иначе `→ /login`.
- Страницы: `LoginPage`, `RegisterPage` (с показом ошибок по полям, авто-логин после регистрации), `ProfilePage` (минимум: имя/телефон/статус email + выход), обновлён `HomePage`.
- Стили форм/кнопок — mobile-first (крупные поля и тап-таргеты ≥48px).

## Почему так

- **Логин по телефону, а не email** — так просил пользователь: телефон уникальный идентификатор. Провайдер Security настроен на `property: phone`.
- **`json_login` без success-handler'а**: в Symfony 7 authenticator логинит (пишет сессию) и, если success-handler не задан, **передаёт управление в контроллер** `check_path`. Поэтому `login()` не заглушка, а реально возвращает пользователя (как `/api/me`). Разобрался опытным путём — сначала думал, что контроллер не должен исполняться.
- **Нормализация телефона на бэке** — нельзя доверять формату с фронта; дубль `+7 900...` и `8 900...` должен ловиться. Проверено тестом.
- **Сессия, а не JWT** — SPA и API на одном домене (BeGet), cookie-сессия работает без CORS и проще/безопаснее. Как в jegebce.
- **`.env.test.local` с базой `nz`** (не `nz_test`) — test-окружение Symfony **не читает `.env.local`**, а Doctrine в тестах сам добавляет суффикс `_test`. Поэтому базовое имя `nz` → фактически `nz_test`.

## Как проверить

Через браузер (открыть **http://localhost:5175/**):
1. «Регистрация» → имя/телефон/пароль (пароль ≥6) → сабмит. Должно залогинить и вернуть на главную с «Привет, Имя».
2. «Личный кабинет» → видно имя/телефон, email «не указан». Кнопка «Выйти» → возврат на главную, снова видны «Войти/Регистрация».
3. «Войти» → тот же телефон/пароль. Неверный пароль → «Неверный телефон или пароль».
4. Открыть `/profile` не залогинившись → редирект на `/login`.

Через API/тесты:
- `make test` → `OK (6 tests)`.
- Тестовый пользователь в dev-БД: телефон `79991234567`, пароль `secret1`, роль `ROLE_ADMIN` (ему доступен http://localhost:8090/admin по сессии).

## Заметки / мелкий долг

- Неавторизованный запрос к `/api/*` отдаёт **HTML** 401 (дефолт Symfony), а не JSON. Фронту не мешает (смотрит на статус-код), но для чистоты API позже добавить JSON entry-point.
- `POST /api/logout` отдаёт **302** (дефолтный редирект). Сессия гасится корректно; при желании навесить handler на 204.
- В админку (`/admin`) сейчас можно попасть только по cookie, полученному через `/api/login` (формы входа в самой админке нет). Для отдельного входа админа — при необходимости добавить `form_login` позже.
