# CLAUDE.md — рабочие инструкции и стандарты кода

Проект **nz** — сайт для турниров по настольному теннису.
Стек: **Symfony 7.4 LTS (backend, JSON API) + React/Vite (frontend), PHP 8.5, MySQL 8**.
План работ: см. `docs/ROADMAP.md`.

Домен/слот на BeGet пока не назначен — в коде проект называется `nz` (легко переименовать позже).

---

## Что за проект (суть функционала)

- **Регистрация игрока** — по номеру телефона (уникальный) + пароль. SMS-подтверждения нет.
- **Личный кабинет** — профиль, можно добавить и подтвердить email (верификация письмом по SMTP).
- **Турниры** — админ создаёт турнир, игроки на него регистрируются.
- **Чекин** — отметка, что игрок пришёл на турнир. Админ может **добавить игрока «с колёс»** (walk-in): завести аккаунт по телефону+имени прямо на месте и сразу чекинить.
- **Жеребьёвка** — одна кнопка у админа: система случайно делит чекнувшихся на 2 стола по 16 и внутри стола случайно составляет пары первого тура.
- **Сетка** — single-elimination на каждом столе: 1/16 (8 матчей) → 1/4 → 1/2 → финал. Победитель матча проходит дальше. Победитель стола становится чемпионом (`isChampion`) и в обычных турнирах больше не участвует (турниры чемпионов — позже, пока не нужны).

## Общие правила работы

- Коммитим/пушим только по запросу пользователя. Работа — в ветке, не в `master`.
- Перед удалением/перезаписью файла — посмотреть, что внутри.
- Не выдавать сделанным то, что не проверено. Тесты упали — так и сказать.
- Открытые/блокирующие вопросы из роадмепа не обходить молча — уточнять.

## Режим работы: одна задача за запуск + дневник

Пользователь запускает меня обычно по одной задаче (`T0.1`, `T1.2`, ...) из `docs/ROADMAP.md` за раз, проверяет результат позже. Поэтому:

- **Брать одну задачу за запуск** (если не сказано иное), довести до проверяемого результата, отметить `[x]` в роадмепе.
- **Каждая задача = отдельный коммит** (по запросу) + **запись в дневнике** `docs/devlog/YYYY-MM-DD-<task-id>.md`.
- Формат записи дневника:
  1. **Что сделал** — список изменений простыми словами.
  2. **Почему так** — каждое решение с обоснованием: «сделал X, потому что Y, а не Z потому что...».
  3. **Как проверить** — пошагово, что нажать/открыть/ввести, чтобы убедиться что работает.
  4. **Что дальше** — следующая задача из роадмепа.

### Пользователь НЕ знает React (и слабо — современный бэк)

Объяснять **как в детском саду**, особенно фронтенд:
- Любой новый термин/концепт React (компонент, хук, пропсы, state, JSX, роутинг) — пояснять при первом появлении 1–2 предложениями человеческим языком, можно с бытовой аналогией.
- Не считать ничего «очевидным». Завёл новый файл/папку/команду — сказать зачем она и что делает.
- Show, don't assume: давать конкретные команды и что должно появиться на экране.

## Безопасность (критично)

**НЕЛЬЗЯ:**
- Хардкодить секреты (DB-пароли, API-ключи, SMTP) в коде или коммитить в git. Только `.env.local` / Symfony secrets vault.
- Хранить/сравнивать пароли в открытом виде. Только `password_hasher` (argon2id).
- Конкатенировать пользовательский ввод в SQL. Только Doctrine / параметризованные запросы.
- Эхать ввод в ответ без экранирования (XSS).
- Самопальная аутентификация. Только Symfony Security.

**НУЖНО:**
- Все секреты — через env / vault, в `.gitignore`.
- Валидация входных данных на бэке (Symfony Validator / `#[Assert\*]`), не доверять фронту.
- CSRF-защита на мутациях, CORS настроен явно под нужный origin.
- Cookie сессии: `HttpOnly`, `Secure`, `SameSite`.
- Телефон нормализовать перед сохранением (единый формат), проверять уникальность на уровне БД (unique constraint) и Validator.

## Локальная разработка — только Docker

- Хост-PHP/composer/node **не используем**. Всё в контейнерах (`compose.yaml`): `php` (8.5-fpm), `nginx` (docroot `backend/public`, сайт на http://localhost:8090), `db` (MySQL 8, хостовый порт 3308), `node` (React, Vite на 5175).
- Команды — через `Makefile`: `make up`, `make sh`, `make composer c="..."`, `make console c="..."`.
- PHP-команды запускать **внутри контейнера под пользователем `dev`** (uid 1000), чтобы файлы не были root.
- Прод (BeGet shared) Docker не использует — там обычный PHP-FPM. Docker лишь повторяет стек для паритета.

## Backend — PHP / Symfony 7.4

- **PHP 8.5**, `declare(strict_types=1);` в каждом файле.
- Стандарт **PSR-12**. Статанализ PHPStan (level max или близко).
- Типизировать всё. Использовать enums, readonly, named args, constructor property promotion. `final` для классов по умолчанию.
- Архитектура: тонкие контроллеры, бизнес-логика в сервисах (жеребьёвка, продвижение по сетке — отдельные сервисы). DI через autowiring.
- Doctrine: атрибуты для маппинга, миграции через `doctrine/migrations` (схему руками не правим).
- API: ответы JSON через Serializer, единый формат ошибок. HTTP-коды по смыслу (200/201/400/401/403/404/422).
- Именование: классы `PascalCase`, методы/переменные `camelCase`, эндпоинты в URL `kebab/snake` (`/api/tournaments`).
- Один PHP-файл = один класс, namespace по PSR-4 (`App\...`).

## Кэш и логи

- `backend/var/cache/` и `backend/var/log/` — в git не коммитим.
- После изменения конфигурации чистить кэш: `php bin/console cache:clear`.
- Логи — только через PSR-3 `LoggerInterface`. Никаких `error_log()`/`echo`/`file_put_contents` в лог руками.

## Базы данных и миграции

СУБД — **MySQL 8** (локально контейнер `db`, прод — MySQL на BeGet).

- Маппинг атрибутами Doctrine; схему руками не правим, только миграции.
- Миграцию — на **каждое** изменение схемы (`make console c="make:migration"`), применять `doctrine:migrations:migrate`.
- **Проверять сгенерированную миграцию перед коммитом.**
- **Не редактировать смерженные миграции.** Нужна правка — новая миграция сверху.
- MySQL **не** поддерживает `IF NOT EXISTS` для `ADD COLUMN`/`CREATE INDEX` — не закладываться.

## API — документация

- OpenAPI/Swagger через PHP-атрибуты (`nelmio/api-doc-bundle` + `#[OA\...]`), рядом с контроллером.

## Frontend — React / Vite

- **Mobile-first.** Основная аудитория заходит с телефона (регистрация, чекин, отметка результата, просмотр сетки). Верстать сначала под мобилу, потом расширять на десктоп. Крупные тап-таргеты, минимум мелкого текста, сетку делать читаемой на узком экране.
- **Функциональные компоненты + хуки**, без классовых. TypeScript, типы для API-ответов.
- Линт `oxlint`.
- Компоненты `PascalCase`, файл под компонент. Логику запросов вынести (хуки/сервис-слой), не дёргать `fetch` россыпью.
- Приватные роуты (личный кабинет, админ-действия) — через guard, опираясь на `GET /api/me`.

## Тесты

- Backend: PHPUnit. Обязательно покрывать: auth, жеребьёвку и продвижение по сетке (критичная логика).
- Перед «готово» — прогнать тесты и линтеры, показать результат.

## Git

- Ветка под фичу от `master`. Коммит-месседжи и PR — только по запросу.
- **Сообщение коммита — только короткое название одной строкой.** Без тела, без списков изменений.
- **Никогда** не добавлять `Co-Authored-By` или любые приписки о Claude/ассистенте — ни в коммиты, ни в PR.

## Коммуникация с агентами
